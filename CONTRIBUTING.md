# Contributing to iUSB

Thank you for considering a contribution. iUSB exists to make repair shops faster and customers better informed; every reproducible bug report, every compatibility note, and every PR that lands a fix helps that mission.

## During private beta (now)

Source code lives with early-access shops under MIT license; the public source release is scheduled for Q3 2026. Until the source lands, the most useful contributions are:

1. **Scanner compatibility reports** when iUSB fails on a specific OS, distro, or hardware combination. Use the [Scanner Compatibility](.github/ISSUE_TEMPLATE/scanner_compatibility.md) issue template.
2. **Bug reports** for early-access shops running the beta. Use the [Bug Report](.github/ISSUE_TEMPLATE/bug_report.md) template. Attach the JSON output (`launcher --json`) if you can; redact anything sensitive first.
3. **Feature requests** grounded in a concrete shop workflow. Use the [Feature Request](.github/ISSUE_TEMPLATE/feature_request.md) template. The best feature requests describe the workflow problem first and propose the feature second.
4. **Documentation pull requests** against anything in `README.md`, `docs/`, or `examples/`. These are very welcome right now.

## After public release

Once `src/` ships, contributions to the scanner code itself become possible. A few principles to know up front:

### Code style

- **Bash** scripts follow [Google's Shell Style Guide](https://google.github.io/styleguide/shellguide.html) and pass `shellcheck` clean.
- **PowerShell** scripts pass `PSScriptAnalyzer` clean with the default rule set.
- **Python** (used only in tooling, not in the scanner runtime) follows PEP 8 and passes `ruff` clean.
- **HTML** in the report generator is generated by the renderer; do not hand-edit the output.

### What we will and will not merge

We will happily merge:

- Fixes for confirmed bugs with a reproduction
- Compatibility patches for distros and hardware we don't yet support
- Documentation improvements
- New module proposals that ship behind an opt-in flag for the first release
- Test cases that cover existing modules

We will probably not merge:

- New runtime dependencies in the Community edition. iUSB is built on native OS tools by design.
- Telemetry, analytics, or any network calls in Community-edition scan paths.
- Modules that require admin or root rights without an opt-in elevation prompt.
- Breaking changes to the report HTML structure or `config.json` schema without a migration path.

### Commit and pull request style

- Commit messages: imperative mood ("Fix battery health on M3 Max" not "Fixed battery..." or "Fixes battery...")
- One topic per pull request. Smaller PRs land faster.
- Reference the issue your PR closes (e.g. "Closes #42") in the description.
- Include a test or a manual repro for any fix.

### Licensing of contributions

By submitting a pull request, you agree that your contribution is licensed under the MIT License (the same license as the rest of the project). We do not require a separate CLA at this time.

## Reporting good bugs

A great bug report tells us four things:

1. **What you did.** The exact command and any config differences from the default.
2. **What you expected to happen.**
3. **What actually happened.** Include the relevant portion of the launcher's terminal output and, if the report generated, the JSON output.
4. **The environment.** OS, OS version, form factor, architecture, and iUSB version (`launcher --version`).

For scanner compatibility reports specifically, the [Scanner Compatibility template](.github/ISSUE_TEMPLATE/scanner_compatibility.md) walks you through the right fields.

## Community standards

This project follows the [Contributor Covenant 2.1](CODE_OF_CONDUCT.md). Be kind, assume good faith, and remember the audience is fellow repair techs trying to do their job.
